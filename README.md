# Система учета оборудования с QR-кодами

Веб-приложение для учета оборудования, контроля статусов и отправки заявок на ремонт в Яндекс Трекер.

---

## Описание проекта

Каждому устройству присваивается уникальный QR-код.  
При сканировании QR открывается карточка устройства, где можно посмотреть:

- текущий статус;
- ответственного;
- историю ремонтов;
- признак, забрал ли устройство системный администратор.

Система позволяет создавать заявки на ремонт, которые автоматически отправляются в Яндекс Трекер.

---

## Цели проекта

- Централизовать учет оборудования.
- Упростить доступ к информации по устройствам через QR-коды.
- Сделать процесс ремонта прозрачным и контролируемым.
- Организовать единый канал заявок через Яндекс Трекер.

---

## Технологический стек

- **Frontend:** React + TypeScript + Vite
- **QR-сканирование:** html5-qrcode
- **Backend:** Python + FastAPI
- **База данных:** PostgreSQL
- **ORM и миграции:** SQLAlchemy + Alembic
- **Авторизация:** JWT (для внутренних пользователей)
- **Интеграция заявок:** Яндекс Трекер API
- **Инфраструктура:** Docker Compose (`docker-compose.yml` + **`Dockerfile`** — разработка; `docker-compose.prod.yml` + **`Dockerfile.prod`** — прод: Postgres без порта наружу, статический фронт в nginx, бэкенд на `127.0.0.1:8001`, фронт на `127.0.0.1:8080`). Черновик хостового nginx: `deploy/nginx-fixmykit.example.conf`. Пример переменных: `.env.production.example`.

---

## Продакшен

1. Скопируйте `.env.production.example` → `.env.production`, заполните секреты; задайте `CORS_ORIGINS` (URL фронта) и `VITE_API_BASE_URL` (URL API).
2. Сборка и запуск: `docker compose -f docker-compose.prod.yml --env-file .env.production up -d --build`.
3. На хосте настройте nginx по примеру `deploy/nginx-fixmykit.example.conf` (два `server_name`, прокси на `127.0.0.1:8080` и `:8001`).

Бэкенд: если `CORS_ORIGINS` не пустой, включается `CORSMiddleware` с перечисленными origin (без cookies, `Authorization` разрешён через `allow_headers=["*"]`).

Фронт: при сборке в образ подставляется `VITE_API_BASE_URL`; в рантайме запросы идут на этот хост. Если переменная пустая — по-прежнему относительные пути `/api/...`.

## Функционал (MVP)

### 1) Учет оборудования

- карточка устройства:
  - инвентарный номер;
  - название;
  - категория;
  - серийный номер;
  - кабинет;
  - ответственный.

### 2) QR-коды

- генерация QR-кода для каждого устройства;
- переход в карточку устройства по сканированию QR.

### 3) Карточка устройства

- просмотр текущего ремонтного статуса:
  - `not_in_repair` — не в ремонте;
  - `in_repair` — в ремонте;
- отображение признака:
  - `taken_by_sysadmin` — забрал ли устройство системный администратор (да/нет);
- просмотр истории ремонтов.

### 4) Ремонтные заявки

Заявку может создать:

- авторизованный пользователь;
- неавторизованный пользователь (через публичную форму).

### 5) Публичная форма заявки (без авторизации)

Пользователь указывает:

- Фамилия и Имя; 
- описание проблемы.

Заявка привязывается к конкретному устройству (через QR или инвентарный номер).

### 6) Ответственный по умолчанию

- для всех ремонтных заявок по умолчанию назначается **системный администратор**.

### 7) Интеграция с Яндекс Трекером

- автоматическое создание тикета;
- сохранение в системе:
  - `ticket_id`;
  - `ticket_key`;
  - ссылки на тикет.

### 8) Поиск и фильтрация

- по инвентарному номеру;
- по категории;
- по кабинету;
- по ответственному;
- по ремонтному статусу.
